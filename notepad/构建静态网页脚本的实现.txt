目标是这样的：
	每天我可能会记录一些日志，笔记，这些文本要生成网页格式推送到server，形成博客。
这些静态网页具有相同的画框，只需要贴一些相片和标题进去就行。因此“相框”的重复工作
就交给脚本来完成，定期执行生成具有特定格式名字的HTML文件。

该如果实现呢?
	考虑以下几个问题需要解决
	1. 文字来源
	2. 文字的存储
	3. 存储的格式
	4. 如何贴图
	5. 文件名的格式

解决方案：
	1. 文字来源
	这个按照网站的设定，如果从创作者角度来讲，来源只有我自己，这是板上钉钉的事实。
其次应该是知识的来源；依照个人的生活方式，主要来自：发神经的想法，有时候高大上的
称呼叫灵感，与人沟通的收获，生活中的阅历，读书的摘要，心得和实践。技术的学习，积累
和经验；打酱油的文学创作；科学界的新闻，技术，进展，科普。大致就这些了！

	2. 文字的存储
	文字存储方式这里不是讨论的存储媒介，而是按照什么格式存储，方便网页的生成。
	想了几种方式:xml, json, db, 自定义；xml不了解，暂时忽略。
	最方便的方式是采用json的格式最好。如此，解析起来可以直接使用json库。
某些字段可以省略，也容易扩展。content内容中如果有字符串，转义一下即可。
	如果采用db方式，将会尝试使用sqlite3。增删查改要方便很多。
	
	3. 存储的格式
	3.1 如果采用json，构思一下格式大致如下：
		{
			"type":"技术"
			"title":"构建静态网页的脚本",
			"subtitle":"实现方案解析",
			"date":"2017-05-10",
			"time":"20:51",
			"content":"不拉不拉不拉不拉。。。",
			"sign":"于深圳"
		}

	3.2 如果采用sqlite
	创建两个表
	文章索引列表
		blog_index_tb(
			id, //全局id
			type, 
			class,
			title, 
			subtitle, 
			abstruct, 
			content_id, 
			sign, 
			date, 
			time, 
			modify_time
		);
	文章内容表
		content_tb(id, content);
	type映射表
		//博客分类
		type_name_tb(
			type,
			name
		);
		//详细学科和书籍分类
		class_name_tb(
			type,
			name
		);
			
	详细分类见备注

	文长索引和文体分开，这样设计将有助于查询速度。更重要是为另外一个系统
	做兼容。卖个关子！
	两个分类表，是为了后期博客增删新类而相互独立。class_name_tb只增不会减
	
	3.3 自己设计
	这个重造轮子的工程，闲暇再玩吧。

	3.4还有一种可能就是使用现成的生成工具，对于我来说没兴趣。
	
	这里优先考虑使用sqlite，原因是
		1. 可扩展性更强，编程自由度更大些；
		2. 为了后续工作准备
		3. 后期实现增删查改的前端化
		4. 已有的文件分类列表

	4. 如何贴图
	只要上面的数据准备到位了，这一步就很简单了，将对应数据组合起来，并输出
到文件就完成了。另外值得一提的是数据的获取。获取数据的周期应该具有几个粒度
可选：天，周，月，年。
	天：每天更新网页，依据分类更新到对应目录，插入blog页最新当中,页头
	周：获取一周的博客，生成周报；
	月：获取一月的博客，生成月报；
	年：获取一年的博客，生成年报；
	为了服务这一工作，需要固定的文件名格式。

	5. 文件名的格式
		date_type_id.html
		date：20170510_
		type：read/science/scope/literature/
		id：全局唯一id
